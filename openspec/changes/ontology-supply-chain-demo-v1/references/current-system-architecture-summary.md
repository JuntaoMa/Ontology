# 当前系统架构设计总结

## 文档目的

本文用于总结当前已经形成共识的系统架构设计，作为后续继续修改 `proposal`、`design`、`specs` 的讨论基线。

本文总结的是“当前倾向的架构”，不是最终冻结版本。后续若需要调整，应优先回写到正式 OpenSpec 文档中。

## 一、系统定位

当前系统被定义为一个 **基于本体的智能数据管理与业务问答/分析/规划系统**，核心不是 OWL 式形式推理，而是参考 Palantir 对 Ontology 的理解，将本体作为组织的运行时语义层与行动层。

当前定位可以概括为：

- 本体负责表达业务世界、能力边界和治理边界。
- 智能系统负责在本体约束下完成语义理解、分析、规划和解释。
- Demo 阶段优先展示“问答 + 规划闭环 + 可解释性”，而不是平台化的全量能力。

一句话定义：

> 本体是企业运行时语义与行动模型；智能系统在该模型之上完成问题理解、业务分析、计划生成与重规划。

## 二、总体架构

当前系统的总体架构倾向于采用“本体核心 + 运行上下文 + AI 工作台”的分层结构。

```text
┌──────────────────────────────────────────────┐
│               用户 / Web Demo                │
├──────────────────────────────────────────────┤
│ 对话输入 │ 证据链解释 │ 计划展示 │ 重规划差异 │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│              AI Orchestration Layer          │
├──────────────────────────────────────────────┤
│ Semantic Compiler │ Analysis Engine │ Planner│
│ Explanation Service │ Capability Resolver    │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│              Ontology + Runtime              │
├──────────────────────────────────────────────┤
│ 语义层 │ 能力层 │ 运行层 │ 治理层             │
└──────────────────┬───────────────────────────┘
                   │
                   ▼
┌──────────────────────────────────────────────┐
│             Data / Execution Layer           │
├──────────────────────────────────────────────┤
│ 图存储 │ 关系型业务数据 │ 仿真/规则引擎 │ 代码实现 │
└──────────────────────────────────────────────┘
```

## 三、本体分层设计

### 1. 语义层

语义层用于定义业务世界“是什么”，当前倾向包含：

- `ObjectType`
- `PropertyType`
- `LinkType / RelationType`
- `Interface`
- `Metric`
- `Constraint`
- `Dynamic Security`

这一层用于表达：

- 业务对象及其属性
- 对象之间的业务关系
- 共享抽象能力
- 指标和约束
- 哪些对象、动作、字段在什么边界下可见/可用

### 2. 能力层

能力层用于定义系统“能做什么”，当前倾向将以下元素作为一等能力节点：

- `FunctionSpec`
- `ActionSpec`
- `Policy / Guard`
- 必要时补充 `WorkflowTemplate`

这层参考 Palantir 的 `Function` 和 `ActionType` 思路，但当前更强调“语义契约”而不是“代码本体”：

- `FunctionSpec`：表达业务分析、计算、仿真、排序等逻辑能力
- `ActionSpec`：表达合法业务动作、前置条件、权限、可能的 side effects

当前共识是：

- `Function` 更偏纯逻辑、可解释、可复用、尽量无副作用
- `Action` 更偏业务操作、流程控制、权限治理和副作用管理

### 3. 运行层

运行层用于表达业务系统当前“处于什么状态”，当前倾向包含：

- `ObjectInstance`
- `ObjectSet`
- `Event`
- `StateSnapshot`
- 后续可扩展 `ActionExecution`、`ActionLog`

当前更倾向于将 `Plan / CandidatePlan / ReplanDelta / BusinessIntent` 视为 **AI 工作台中的工作产物**，而不是本体核心概念本身。也就是说：

- 本体负责提供世界模型、规则和合法动作
- 智能系统在本体约束下生成计划和分析结果

这一点后续仍可调整，但当前更接近 Palantir 风格。

## 四、AI 层职责划分

当前 AI 编排层采用单编排器模式，不做多 Agent 协作。逻辑边界如下：

- `Semantic Compiler`
  - 将自然语言问题转换为 `BusinessIntent`
  - 基于本体进行语义对齐，得到 `OntologyContext`
- `Capability Resolver`
  - 根据命中的对象类型、接口、目标输出类型发现相关 `FunctionSpec / ActionSpec`
- `Analysis Engine`
  - 调用只读能力完成查询、推导、仿真、排序，并产生结构化证据链
- `Planner`
  - 仅在本体已声明的 `ActionSpec` 范围内生成计划与重规划结果
- `Explanation Service`
  - 将本体语义、原始业务数据、派生指标和计划步骤汇总为可解释输出

当前明确限制：

- LLM 不直接决定业务真实性
- LLM 不直接生成未经约束的动作
- LLM 主要负责语义压缩、候选召回、语言组织和解释生成

## 五、Function / Action 的表达与存储

### 1. 不直接以代码作为事实源

当前共识是不将 Python/TypeScript 代码本身作为本体中的主要内容，而是将 `Function/Action` 存储为 **结构化能力契约**，代码只作为实现引用。

### 2. 结构化 IR 作为中间表示

当前倾向引入 `结构化 IR（Intermediate Representation）` 作为统一事实源：

```text
自然语言 / 表单 / 可视化编排
            ↓
        结构化 IR
            ↓
   校验 / 解释 / 检索 / 编译
            ↓
 Python / TypeScript / SQL / 仿真引擎 / 工作流引擎
```

IR 的目标是把用户意图转换为：

- 可校验
- 可版本化
- 可解释
- 可编译执行

的业务契约。

### 3. Function IR 与 Action IR 分开设计

当前倾向不使用一种通用语言描述所有能力，而是：

- `Function IR`
  - 重点描述输入输出、依赖对象、逻辑类型、计算图、是否纯函数
  - 适用于查询、规则、公式、仿真、排序
- `Action IR`
  - 重点描述输入、权限、前置条件、步骤、调用的函数、输出、副作用、审批
  - 适用于分析任务、业务变更、流程编排

### 4. 用户交互方式

当前推荐的人机协作方式不是“用户直接复核生成代码”，而是：

```text
自然语言起草
   ↓
LLM 生成 IR 草案
   ↓
业务摘要视图 / 流程图 / 规则表 / 示例测试
   ↓
用户复核与调整
   ↓
必要时映射到高码实现
```

也就是说：

- 自然语言用于起草
- 结构化 IR 是唯一事实源
- 图形/表格/示例是主要复核界面
- 高码是高级逃生口，不是默认审核面

## 六、Function / Action 与本体的关系建模

当前倾向将 `FunctionSpec / ActionSpec` 作为本体图中的一类 **能力节点**，但不与海量业务实例直接常驻相连，而是主要连接到类型层和接口层，形成异构图。

示意结构如下：

```text
Ontology Heterogeneous Graph
├─ ObjectType / PropertyType / LinkType / Interface
├─ FunctionSpec / ActionSpec / Policy
└─ ObjectInstance / ObjectSet / Event / Snapshot
```

建议的关键边类型包括：

- `APPLIES_TO`
- `READS_TYPE`
- `WRITES_TYPE`
- `PRODUCES`
- `REQUIRES`
- `GUARDED_BY`
- `CALLS`
- `IMPLEMENTED_BY`

当前重点原则：

- 能力节点优先连接 `ObjectType / Interface / OutputType`
- 不将能力节点直接常驻连接到大量实例节点
- 图中存能力契约与关系，不存大段代码正文

## 七、能力发现与上下文控制

为了避免能力数量增长后难以检索和上下文爆炸，当前倾向采用分阶段召回：

### 1. 类型约束召回

先根据问题命中的：

- `ObjectType`
- `Interface`
- `OutputArtifact`

做硬过滤。

### 2. 图邻域扩展

从命中的类型节点向外扩展 1-2 跳，仅获取相关 `FunctionSpec / ActionSpec`。

### 3. 契约兼容性过滤

过滤掉：

- 输入不满足的能力
- 当前权限不可用的动作
- 当前场景不需要的写操作
- 已废弃或实验性的能力

### 4. 语义重排

最后才使用 embedding 或 LLM 做 rerank，而不是直接对全库能力做向量搜索。

### 5. 分级上下文

每个能力维护两种表示：

- `Full Spec`：供系统和开发者使用
- `Capability Card`：供 LLM 与 UI 快速使用的压缩卡片

这用于控制上下文长度和解释复杂度。

## 八、数据与存储设计

当前倾向的数据层分工如下：

- 图存储
  - 保存本体语义层与能力层
  - 维护类型关系、能力关系、解释元数据
- 关系型存储
  - 保存模拟业务数据、案例数据、派生分析结果
- 执行层
  - 承载公式、规则、仿真、排序、工作流运行和代码实现引用

当前仍坚持：

- v1 业务系统全模拟
- v1 本体只读
- v1 不接真实 ERP/WMS/TMS

## 九、用户体验与 Demo 形态

当前 Demo 仍以技术评审为受众，采用三联屏解释式界面：

- 左侧：对话、问题上下文、限制条件、默认假设
- 中间：本体命中项、函数、指标、证据链、解释卡片
- 右侧：计划步骤、依赖关系、预期影响、重规划差异

主链路保持不变：

1. 用户提出业务问题
2. 系统语义对齐到本体
3. 系统执行分析并返回证据链
4. 用户要求生成方案
5. 系统在合法动作边界内规划
6. 用户追加约束，系统重规划

## 十、当前最重要的开放方向

当前架构中仍有几个值得继续收敛的问题：

- `Plan` 是否长期留在 AI 工作台层，还是后续进入本体运行层
- `ObjectSet` 是否要在 v1 就被提升为一等概念
- `Interface` 是否要在 v1 正式进入建模与检索链路
- `Constraint` 与 `Dynamic Security` 的边界如何进一步固定
- 制造业“设计变更切换成本分析”场景是否作为下一阶段核心样例

## 十一、当前架构结论

截至目前，系统架构的核心结论可以压缩为：

1. 本体不是静态知识库，而是运行时语义与行动模型。
2. 本体应分为语义层、能力层、运行层和治理层，而不是只建对象关系。
3. `Function/Action` 应作为能力节点进入异构图，但存储为结构化语义契约，而不是直接存代码。
4. 智能系统应围绕本体做语义对齐、能力发现、分析、规划和解释，而不是绕开本体直接回答。
5. 用户友好的能力作者方式应采用“自然语言起草 + IR 中间层 + 多视图复核 + 高码逃生口”的组合。
