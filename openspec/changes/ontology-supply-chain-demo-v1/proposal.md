## Why

当前仓库尚未沉淀一套可执行的系统方案，但目标已经比较明确：需要做一个面向内部技术评审的供应链/运营 Demo，用来展示“基于本体的智能数据管理与业务问答/规划闭环”是否具备工程可行性与展示说服力。现在写入正式变更文档，可以先把场景、边界、能力和后续实现顺序固定下来，避免后续讨论持续停留在概念层。

## What Changes

- 新增一套面向供应链/运营场景的运行时本体能力，覆盖 `object`、`relation`、`metric`、`function`、`constraint`、`action` 的静态与动态层定义。
- 新增一条“自然语言问题 -> 语义对齐 -> 查询分析 -> 计划生成 -> 约束变化后重规划”的业务闭环，并要求每一步都可解释。
- 新增一组只读本体接口，用于查询对象语义、可执行动作、函数说明和规划约束，不支持在线修改本体。
- 新增一套模拟业务系统与样例数据，用于伪装库存、在途、采购、需求和风险信号，保证 Demo 稳定可演示。
- 新增一个可交互 Web Demo，集中展示问答、证据链、本体语义命中、计划步骤和重规划结果。
- 新增一套针对 Demo 的验收标准，重点验证语义对齐质量、证据完整性、计划合法性和解释能力。

## Capabilities

### New Capabilities
- `ontology-runtime`: 定义供应链 Demo 使用的运行时本体模型、对象语义查询方式和合法动作边界。
- `semantic-qa-planner`: 定义自然语言意图解析、语义对齐、查询分析、计划生成与重规划的系统行为。
- `demo-experience`: 定义用于技术评审的 Web Demo 交互流程、解释方式和展示要求。

### Modified Capabilities
- 无

## Impact

- 影响后续的后端服务边界，包括本体服务、业务数据服务、语义编排器、分析服务、规划服务和解释服务。
- 影响后续的前端 Demo 结构，包括对话区、本体/证据链区、计划区的三联屏布局。
- 影响未来的技术选型与样例数据组织方式，默认会引入图存储、关系型数据存储和 LLM 编排层。
- 为后续实现提供单一事实来源，后续编码、任务拆解和验收都以本变更中的 proposal/design/specs/tasks 为准。
