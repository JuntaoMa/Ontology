## ADDED Requirements

### Requirement: Demo 必须提供三联屏交互界面
系统 MUST 提供一个面向技术评审的 Web Demo，并采用三联屏布局。左侧 MUST 展示对话内容、当前问题上下文与限制条件；中间 MUST 展示本体命中项、函数、指标和证据链；右侧 MUST 展示计划步骤、依赖关系、预期影响和重规划差异。

#### Scenario: 打开缺货分析页面
- **WHEN** 评审人员进入 Demo 主界面
- **THEN** 页面同时呈现对话区、语义/证据区和计划区三类信息容器

### Requirement: Demo 必须支持完整主链路演示
系统 MUST 支持一条预置的端到端演示路径：提出缺货问题、查看分析依据、请求处置方案、追加限制条件并查看重规划结果。该主链路 MUST 可以在不依赖真实外部系统的情况下稳定运行。

#### Scenario: 按主脚本完成一次演示
- **WHEN** 评审人员依次执行“问风险”“看原因”“要方案”“加限制”四个步骤
- **THEN** 系统在单次会话中连续返回一致的分析、解释和计划结果

### Requirement: 解释视图必须揭示语义来源
系统 MUST 允许用户从回答或计划中的任一关键元素跳转到对应的本体解释或证据项。解释视图 MUST 明确区分“本体定义”“原始业务数据”“派生指标/函数结果”三类来源。

#### Scenario: 查看回答中的证据来源
- **WHEN** 用户点击缺货原因中的某个关键指标或动作建议
- **THEN** 系统展示该内容来自哪类语义元素、对应的数据记录和派生逻辑说明

### Requirement: Demo 必须内置多条稳定案例
系统 MUST 预置至少六条可重复演示的案例，其中至少包含三条缺货风险案例、两条非缺货案例和一条边界模糊案例。用户 MUST 能够从界面中切换预置案例或使用默认主案例快速进入演示。

#### Scenario: 切换到非缺货案例
- **WHEN** 用户从默认主案例切换到一个库存充足的案例
- **THEN** 系统展示无缺货结论，并保持相同的语义解释与计划入口结构
