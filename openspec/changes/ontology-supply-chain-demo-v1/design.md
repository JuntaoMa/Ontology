## Context

当前仓库只有 OpenSpec 基础目录，尚未存在业务代码或现成的数据模型，因此这次变更的第一目标不是补充零散实现，而是把一个可执行的 Demo 方案先固化成工程约束。Demo 面向内部技术评审，场景固定在供应链/运营，目标是在 `2-4 周` 内完成一条稳定主链路：

1. 用户询问某仓某 SKU 是否会在给定时间窗内断货。
2. 系统基于本体识别对象、关系、指标、函数和约束，并查询模拟业务数据。
3. 系统给出分析结论与证据链，并解释背后的本体语义。
4. 用户继续要求生成处置方案，系统在合法动作边界内生成计划。
5. 用户增加新限制后，系统对既有计划进行重规划。

本体采用接近 Palantir 的运行时语义模型，而非 OWL 推理体系。重点在于让系统能够把本体作为“业务语义与可行动作的运行时上下文”来使用，而不是做通用知识表示研究。

## Goals / Non-Goals

**Goals:**
- 固定一个最小但完整的供应链本体范围，足以支撑问答、分析、规划和重规划。
- 固定一个稳定的系统架构，让本体、模拟业务数据和 LLM 编排职责清晰分离。
- 让问答与规划共享同一份语义上下文，避免问答和规划各做一套逻辑。
- 让所有回答和计划都可追溯到本体定义、业务数据和派生指标，便于技术评审。
- 将 Demo 范围约束在只读本体、全模拟业务系统和单编排器模式，降低交付风险。

**Non-Goals:**
- 不实现 OWL/RDF 兼容、描述逻辑推理或通用知识图谱平台能力。
- 不接入真实 ERP、WMS、TMS 或审批系统。
- 不支持在线修改本体、版本治理、审批流或多租户隔离。
- 不实现多 Agent 自治协作或通用工作流引擎。
- 不追求行业通用性，v1 只覆盖供应链/运营的单一演示主线。

## Decisions

### 1. 以单场景闭环替代通用平台展示

选择“缺货风险分析与处置规划”作为唯一主场景，而不是做通用平台壳子。原因是内部技术评审更关注语义建模是否能真正驱动问答与规划，而不是抽象概念是否完整。单场景闭环更容易展示对象、函数、动作、约束和重规划之间的联动。

备选方案是做“通用平台 + 多业务域示例”。该方案展示面更广，但在当前仓库没有基础实现的前提下，交付风险高，且难以在 2-4 周内保证演示质量。

### 2. 本体采用运行时语义模型而非 OWL

本体静态层固定包含 `ObjectType`、`RelationType`、`Metric`、`Function`、`Constraint`、`ActionType`；动态层固定包含 `ObjectInstance`、`Event`、`StateSnapshot`、`Task`、`Plan`、`ActionExecution`。这些模型必须直接服务于：

- 自然语言问题的语义对齐；
- 业务数据到对象实例的映射；
- 规划时动作合法性的约束；
- UI 对语义和证据的解释展示。

备选方案是使用 OWL/RDF 和现成推理机。该方案在学术表达上更完整，但与本次 Demo 的“运行时业务语义”和“可执行动作约束”不完全匹配，工程复杂度也更高。

### 3. 架构采用单编排器 + 六个服务边界

后续实现固定为以下边界：

- `Ontology Service`：提供本体定义、语义查询、对象/动作/函数解释。
- `Business Data Service`：提供模拟库存、在途、采购、需求和风险信号数据。
- `Semantic Compiler`：把自然语言问题转换成 `BusinessIntent` 与 `OntologyContext`。
- `Analysis Engine`：执行查询与派生指标计算，生成答案草稿和证据项。
- `Planner`：在本体声明的 `ActionType` 范围内生成计划与重规划结果。
- `Explanation Service`：整合本体语义、业务数据和派生指标，输出给前端展示。

这些能力可以在物理部署上合并实现，但逻辑边界必须保留。编排层采用单编排器，而不引入多 Agent 协作，以减少状态管理和调试复杂度。

### 4. LLM 只负责语义压缩和语言生成，不直接决定业务真实性

LLM 的职责固定为：

- 从自然语言中抽取目标、对象、时间窗、过滤条件和限制条件；
- 结合本体候选项完成语义对齐；
- 将结构化结果翻译成自然语言解释；
- 为计划步骤生成自然语言说明。

LLM 不能直接决定库存是否充足、动作是否合法、指标如何计算。这些结论必须来自本体声明、模拟业务数据和确定性函数。

### 5. 数据层采用“图语义 + 关系数据 + 固定样例”

后续实现默认采用如下数据分工：

- 图存储保存本体类型、关系、动作、函数与解释元数据。
- 关系型存储保存模拟业务数据和派生分析结果。
- 固定样例数据覆盖至少 `3 个缺货案例`、`2 个非缺货案例`、`1 个边界模糊案例`。

这样设计的目标是同时满足语义浏览、结构化查询和稳定演示。备选方案是全部使用单一文档存储，但会降低语义查询和实体关系表达的清晰度。

### 6. Web Demo 固定为三联屏解释式界面

前端固定包含三个区域：

- 左侧：对话、当前问题上下文、默认假设与限制条件。
- 中间：命中的对象、关系、函数、指标、证据链和解释卡片。
- 右侧：计划步骤、依赖关系、预期影响和重规划差异。

这样可以把“回答内容”和“回答依据”同时展示出来，避免 Demo 退化成单纯聊天。

## Risks / Trade-offs

- [本体建模过宽导致实现拖延] -> 将 v1 对象范围固定在 `Product`、`Warehouse`、`InventoryPosition`、`Shipment`、`PurchaseOrder`、`Supplier`、`DemandSignal`、`RiskAlert`，新增范围需另行评估。
- [LLM 输出与本体术语不一致] -> 采用候选项约束的语义对齐流程，先生成结构化意图再生成自然语言。
- [规划结果看起来合理但不可执行] -> 规划器只允许使用已声明的 `ActionType`，并强制输出前置条件与预期影响。
- [Demo 样例过少导致演示单薄] -> 固定准备多条案例路径，并在 UI 中支持切换预置场景。
- [全部模拟降低真实感] -> 在设计文档中明确这是为了展示语义链路与架构边界，真实系统集成不属于 v1 目标。

## Migration Plan

1. 先完成 OpenSpec 文档，作为唯一设计基线。
2. 再实现本体与模拟数据层，产出可复用的种子数据。
3. 然后实现语义对齐、分析和规划接口，并以接口契约驱动前端联调。
4. 最后补齐前端演示、验收样例和讲解脚本。

本变更落地前没有旧系统需要迁移，因此不存在数据迁移和回滚问题；若后续需要改变演示范围，应通过新的 OpenSpec 变更调整，而不是直接偏离本设计。

## Open Questions

当前版本没有阻塞实现的开放问题。若后续要接入真实业务系统、支持本体写入或扩展到多业务域，应单独提出新的变更，而不是在本变更内隐式扩展。
