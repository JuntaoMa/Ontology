## ADDED Requirements

### Requirement: 查询工具必须与业务计算分离
系统 MUST 将查询工具限制为数据定位、路径发现、查询规划、Cypher 生成和结果绑定，不得在查询工具内部隐式完成业务计算、打分、推荐或成本结论。

#### Scenario: 查询新物料属性后不直接给出准备周期
- **WHEN** 智能体使用查询工具读取新物料的属性、供应方式和采购参数
- **THEN** 查询工具只返回这些数据及其证据绑定，不直接返回“新物料准备周期”的计算结果

### Requirement: 系统必须定义统一的 Query IR
系统 MUST 定义一套面向设计变更场景的 Query IR。Query IR MUST 至少包含 `intent`、`anchors`、`steps`、`return` 和 `evidence` 五部分，并支持锚点定位、多跳路径、字段过滤和结果绑定。

#### Scenario: 为设计变更问题生成查询计划
- **WHEN** 智能体需要回答“ECN-204 影响了哪些批次”
- **THEN** 系统先形成包含变更锚点、路径步骤和返回绑定的 Query IR，而不是直接生成自由文本查询语句

### Requirement: Query IR 必须可以落到 Cypher 或 dry-run 说明
系统 MUST 支持将有效的 Query IR 转换为 Cypher 查询说明，或在没有数据库连接时输出 dry-run 级别的查询执行说明。两种模式 MUST 保留相同的查询步骤和证据绑定含义。

#### Scenario: 无数据库连接时查看查询计划
- **WHEN** 当前环境没有可用的 Neo4j 连接
- **THEN** 系统输出基于 Query IR 的查询步骤说明和预期绑定结果结构，而不是伪造数据库结果

### Requirement: 查询结果必须保留证据绑定
系统 MUST 在查询结果中保留对象实例、关系路径、过滤条件和返回字段的证据绑定，以便后续业务函数和最终解释复用。

#### Scenario: 为切换成本计算准备输入
- **WHEN** 查询工具完成批次、库存和到货数据的读取
- **THEN** 系统返回结构化绑定结果，并标明这些结果来自哪些对象、关系或字段
