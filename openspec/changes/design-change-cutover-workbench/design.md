## Context

当前仓库里的制造设计变更切换成本分析 Demo 已经沉淀了场景、对象、函数和动作的最小链路，但这些内容仍主要散落在前端示例代码里，尚未形成一套可供智能体稳定调用的工作台式能力。用户已经明确两项约束：

1. schema 相关内容优先存放在 JSON 文件中，作为静态事实源；
2. skill 中的工具优先采用格式化自然语言文本来定义，而不是优先实现成脚本接口。

同时，真实业务数据与当前 Demo 样例数据不一定一致，因此 skill 中的函数与动作不能直接绑定某一套固定字段或假设；必须把“如何查到数据”和“如何基于数据做计算/编排”分离，以便未来替换 Neo4j 中的实例数据时，计算逻辑仍可复用。

## Goals / Non-Goals

**Goals:**
- 为制造设计变更切换成本分析场景建立一套可直接放入 skill 的工作台设计。
- 将工具分成 `查询工具`、`业务函数`、`业务动作` 三类，并明确边界。
- 用格式化自然语言模板表达每个工具的职责、输入、输出、依赖与限制，保证易读和可维护。
- 为查询工具定义统一的查询 IR，使智能体先形成查询计划，再落到 Cypher 或 dry-run 说明。
- 确保 `新物料准备周期`、`切换成本` 等业务逻辑与具体数据查询步骤显式解耦。

**Non-Goals:**
- 不在本次设计中定义真实 Neo4j 数据库的最终字段标准。
- 不要求所有工具先实现为可执行脚本或 MCP 工具。
- 不覆盖写回 MES、锁批次、执行真实变更等生产执行动作。
- 不扩展到供应链缺货规划等其他业务域。

## Decisions

### 1. Skill 采用“自然语言工具目录 + JSON schema 资产”的组合

skill 主体采用 `SKILL.md + references + assets/schema/*.json` 的形式实现。`SKILL.md` 只保留工作流和调用约束；具体工具定义、QIR 结构、函数契约和动作契约分别放到引用文档和 JSON 资产中。

这样做的原因是：
- skill 本身首先是给另一个智能体阅读的，而不是给程序直接调用的；
- 当前目标是先快速形成稳定、易懂的工作台，而不是先建设完整工具链；
- JSON 资产适合承载 schema、关系、查询模式和契约摘要，自然语言文档适合承载“如何使用这些资产”的方法。

备选方案是直接实现一组脚本工具并在 skill 中调用。该方案在长期更自动化，但当前会把问题过早推向执行接口实现，而不是先统一业务语义。

### 2. 查询工具、函数、动作严格分层

本次 skill 中的能力固定拆成三层：

- `查询工具`：负责定位对象、发现路径、形成 QIR、生成 Cypher、读取实例数据；
- `业务函数`：负责根据已查询到的数据做确定性计算、推导、排序、解释；
- `业务动作`：负责按业务主线串联多个查询工具和函数，形成一个完整分析流程。

这意味着：
- 查询工具不能隐式完成业务计算；
- 函数不能在内部“顺便去查库”；
- 动作不能绕过函数直接输出拍脑袋结论。

备选方案是让单个“超级工具”同时完成查数、计算和回答。该方案实现快，但会迅速失去可解释性与可复用性。

### 3. Query IR 作为查询工具与底层数据访问之间的唯一中间层

查询工具不直接承诺“智能体必须手写 Cypher”，而是先要求形成统一的 `QIR`。QIR 至少包含：

- `intent`
- `anchors`
- `steps`
- `return`
- `evidence`

其中 `steps` 只允许表达查询层关注的操作，例如：
- 解析锚点对象；
- 沿关系做多跳路径扩展；
- 按字段过滤；
- 投影查询结果；
- 绑定后续函数所需输入。

涉及业务计算的步骤不放在查询工具中，而交给业务函数。

备选方案是让智能体直接从业务问题生成 Cypher。该方案灵活，但在 schema 变动、数据差异和解释追踪方面不稳。

### 4. 工具定义使用格式化自然语言模板

每个查询工具、函数和动作都使用固定的自然语言模板描述。推荐模板字段如下：

- 名称
- 用途
- 何时使用
- 必要输入
- 可选输入
- 依赖数据
- 执行步骤
- 输出
- 不负责什么
- 常见失败

这样做可以在不实现脚本的前提下，让 skill 具备接近“工具调用说明书”的效果，并让智能体在不同数据环境下保持相同的调用边界。

### 5. 业务函数围绕设计变更切换成本主链路定义

查询工具与函数集合固定围绕当前 Demo 主链路设计：

- 查询工具负责 `识别设计变更锚点`、`读取变更主数据`、`读取受影响批次事实`、`读取旧件库存与新件供应事实`、`形成 Query IR` 和 `生成 Cypher 或 dry-run 说明`；
- 业务函数负责 `归一化变更影响范围`、`计算新物料准备周期`、`筛选候选切换点`、`仿真单批次切换成本`、`排序候选切换方案` 和 `生成切换成本报告`。

其中 `计算新物料准备周期` 明确只根据已提供的物料属性、供应方式、采购周期、物流周期等输入进行计算，不负责自己去 Neo4j 查询物料属性；`读取受影响批次事实` 和 `读取旧件库存与新件供应事实` 则明确归属于查询工具。

### 6. 业务动作固定先做 dry-run 编排

`create_cutover_cost_analysis` 动作默认只做 dry-run 级别的分析编排，不产生外部写操作。动作需要说明：

- 依赖哪些查询工具；
- 调用哪些函数；
- 在什么条件下终止或要求补充上下文；
- 最终如何组织报告、证据链和推荐结果。

这样既符合当前 Demo 的演示目标，也避免把“执行动作”误导成真实业务写入。

## Risks / Trade-offs

- [自然语言工具定义不如脚本严格] -> 使用固定模板，并在 QIR 与函数输入上要求结构化字段。
- [真实数据与 Demo 样例字段差异较大] -> 所有函数都只依赖显式输入，不依赖固定查询语句或隐藏字段。
- [没有脚本执行会降低自动化程度] -> 当前阶段优先保证语义和边界稳定，后续可按同一契约补脚本。
- [QIR 过于抽象导致落地困难] -> 将 QIR 限定在设计变更场景需要的最小操作集，不追求通用图查询语言。
- [动作链过长导致 skill 文本过重] -> 将详细工具定义拆分到 references，并让 `SKILL.md` 只保留导航与规则。

## Migration Plan

1. 先补齐 proposal、design、specs、tasks，固定本次 skill 的边界。
2. 在 `.codex/skills/` 下创建新的设计变更场景 skill。
3. 把现有 Demo 中的 schema、function、action 提炼为 JSON 资产和自然语言契约文档。
4. 先以文档化 skill 方式投入使用，验证智能体是否能按预期完成查询规划与业务分析。
5. 若后续需要更高自动化，再按同一契约补脚本或 MCP 工具，而不是推翻当前设计。

本次变更不涉及线上迁移，也不存在数据回滚问题；若未来调整为脚本优先模式，应通过新的 OpenSpec 变更处理。
