# 问题分流规则

收到问题后，先不要直接回答。先判断它属于哪一类。

在问题分类之前或同时，要先决定应该进入哪个 domain。
如果当前还不知道 domain，先做 domain 发现，而不是直接读取某个业务 schema。

## 1. 查询类问题

典型形式：
- “当前有哪些实例”
- “某个节点有哪些相关节点”
- “某个属性值是什么”
- “从 A 到 B 有哪些路径模板”

处理方式：
1. 发现并选择 domain
2. 识别锚点对象或目标类型
3. 读取该 domain 的数据模型层
4. 使用统一入口获取路径模板或对象详情
5. 输出查询计划与查询结果

## 2. 已知业务流程类问题

典型形式：
- “帮我分析这个业务问题并给出结论”
- “生成一份分析报告”
- “根据现有能力完成一条业务链路”

处理方式：
1. 发现并选择 domain
2. 拆解问题为 `查询需求 + 计算需求 + 编排需求`
3. 检查该 domain 的 function/action 元数据是否覆盖
4. 如果覆盖，形成显式调用计划
5. 输出报告，并说明用了哪些现有能力

## 3. 能力缺口业务流程类问题

典型形式：
- “分析这个问题”，但现有 function/action 不够
- “当前系统要处理这个业务还缺什么能力”

处理方式：
1. 发现并选择 domain
2. 拆解问题
3. 检查该 domain 的能力覆盖情况
4. 标出缺失的 function/action
5. 为缺失能力生成功能草案和逻辑体占位
6. 给出临时流程方案和待复核项

这里的重点不是假装能做，而是明确：
- 哪些能力已有
- 哪些能力缺失
- 为什么缺失
- 应该怎么补

## 4. 新能力定义类问题

典型形式：
- “帮我定义一个新 function”
- “根据这段自然语言生成一个 action”

处理方式：
1. 发现并选择 domain，或明确这是新 domain 能力
2. 先确定它是 function 还是 action
3. 规范化输入输出
4. 对齐到本体对象、属性、边和已有能力
5. 生成元数据草案
6. 生成逻辑体模板

## 5. 分流后的输出规范

无论属于哪一类，都先输出：
- 问题类别
- 选中的 domain
- 锚点对象或目标对象
- 涉及的本体元素
- 使用的统一入口
- 当前结果属于真实查询、能力匹配、草案设计还是逻辑占位
